# 看看工资 PRD

版本：V1.1  
更新日期：2026-06-20  
当前阶段：可演示的 Web 原型 + 小程序首版包 + 官方招聘站 adapter 快照

## 1. 产品定位

看看工资是一个面向高考生、大学生和早期求职者的升学与职业数据决策工具。它不做学校排名，也不做录取概率预测，而是把学校公开资料、专业去向、企业官网岗位、薪资区间和职业信号聚合成可比较、可追溯、可行动的判断。

一句话：先看专业未来能去哪里、市场愿意付多少钱、哪些企业真的在招，再决定是否投入四年。

## 2. 核心用户

- 高考生和家长：快速理解专业与就业、薪资、学校证据之间的关系。
- 大学生：判断转专业、辅修、实习和作品路线。
- 应届生：用岗位反推能力缺口和可迁移专业背景。
- 老师、咨询师和内容团队：需要可复核的数据入口和报告素材。

## 3. 主要问题

用户不是缺信息，而是缺结构化判断：

- 专业目录公开，但很难和真实岗位、薪资、能力要求对应。
- 企业招聘官网公开，但岗位语言分散，不容易转成专业选择信号。
- 学校就业报告公开，但校级数据、专业级数据、企业名单口径混在一起。
- 薪资信息容易被夸大或误读，需要区分官方薪资、市场估算和学校报告。
- 页面如果平铺所有内容，会变成信息噪音，必须分层展示。

## 4. 产品主线

V1.1 收敛为四个一级入口：

1. 学校入口  
   输入学校，优先展示学校官网、招生网、专业目录、就业网、就业质量报告和校招入口。已收录学校展示专业列表、毕业去向、就业率字段、平均工资字段、年度到校企业和证据状态。未收录学校展示可执行的公开入口任务流，而不是空页面。

2. 岗位雷达  
   输入岗位，按关联强度从内到外展示专业排名。每个结果必须说明关联原因、官网岗位证据、薪资代理区间和可继续核验的企业入口。

3. 公司工资  
   按公司 logo、名称、行业、国内/海外、岗位族群、薪资区间和来源口径展示企业。点击公司进入详情，不把所有岗位挤在首页。

4. 职业信号  
   把岗位、专业薪资、学校证据和官方招聘入口归并成统一 CareerSignal。职业信号用于解释“为什么这个方向值得看、适合哪些专业、有哪些风险、下一步要核验什么”。

## 5. 关键用户流程

### 5.1 学校到专业

1. 用户输入学校名。
2. 系统判断是否已收录。
3. 已收录：展示官方入口、专业、校级证据、就业字段和企业线索。
4. 未收录：展示学校类型策略、公开入口目录、证据采集模板和下一步任务。
5. 用户保存证据或点击岗位雷达继续反查。

验收口径：用户即使输入未收录学校，也能获得下一步行动，而不是只看到“暂无数据”。

### 5.2 岗位到专业

1. 用户输入岗位，例如“产品经理”“审计”“酒店运营”。
2. 系统生成专业关联排名。
3. 每个专业展示关联强度、官方岗位数量、代表企业、薪资代理和风险提示。
4. 用户可跳转企业官网继续核验。

验收口径：雷达结果不能只偏向计算机，必须覆盖商科、金融、酒店、医疗、教育、法学、制造等非计算机方向。

### 5.3 公司到工资

1. 用户按行业或地区筛选公司。
2. 公司卡片展示 logo、薪资区间、岗位族群和样本状态。
3. 点击公司进入详情页，展示岗位列表、薪资口径、偏好专业和官网入口。
4. 用户可比较 2-4 家公司。

验收口径：公司列表必须分页或分区展示，详情信息进入详情区，避免首页拥挤。

### 5.4 职业信号到判断

1. 系统从当前岗位快照、学校证据、专业薪资和官方入口生成信号。
2. 默认展示精选信号，不展示全量噪音。
3. 每条信号有来源、原因、风险、相关专业、相关能力和置信度。
4. 后续 API、Agent、小程序都复用同一信号层。

验收口径：每条信号都必须能追溯到 URL 或明确标记为市场代理口径。

## 6. 数据策略

### 6.1 企业岗位

- 优先使用企业官方招聘站 adapter。
- 当前 live adapter 覆盖 ByteDance、Tencent、Alibaba、Baidu、Meituan、JD、Huawei、Kuaishou、Bilibili、Xiaomi、PDD、Midea、Amazon、IKEA、Unilever、L'Oreal 等已接来源，实际以 `jobDataMeta.sources[].normalizedCount` 为准。
- 官方站未公开薪资时，只能输出市场估算，并在 UI 标记“估”。
- 不能把官网当日返回 0 的入口误写成有岗位样本。

### 6.2 学校资料

- 学校官网、招生网、专业目录、就业网和就业质量报告是优先证据。
- 校级证据和专业级指标分开展示。
- 专业级就业率、平均工资、到校企业没有来源时必须显示“待接入”或“待解析”。
- 用户手动保存的证据需要分信任等级，弱证据不能推进为结论。

### 6.3 薪资

- 薪资来源分为 `official` 和 `market-estimate`。
- 官网薪资可作为岗位样本，但仍需看城市、批次和投递条件。
- 市场估算只作为比较代理，不代表企业承诺。
- 页面必须显示刷新时间。

### 6.4 职业信号

CareerSignal 是 V1.1 的核心领域模型。信号类型包括：

- `job`：来自企业官网岗位。
- `salary`：来自专业薪资代理。
- `school`：来自学校公开就业证据。
- `official-source`：来自企业官方招聘入口。

详细机制见 [signal-system.md](signal-system.md)。

## 7. 页面与信息架构

首页不做营销页，直接进入工具：

1. 顶部品牌和主导航。
2. 学校入口作为默认页。
3. 岗位雷达作为第二主入口。
4. 职业信号用于解释数据机制。
5. 公司工资和行业对比用于横向比较。
6. 移动端和小程序壳使用底部 tab，避免长页面堆叠。

## 8. 非目标

V1.1 不做：

- 录取概率预测。
- 学校排名推荐。
- 付费咨询闭环。
- 黑箱人格测评。
- 非官方招聘聚合站替代企业官网。
- 未授权复制第三方长文本内容。

## 9. 成功指标

用户价值：

- 用户 1 分钟内理解产品能解决什么问题。
- 用户 3 分钟内能完成一次“学校 -> 专业 -> 岗位/薪资”的粗筛。
- 用户能看到每个结论背后的证据状态和风险提示。
- 用户能获得下一步核验动作。

数据质量：

- 每条 CareerSignal 有来源、原因、风险和置信度。
- 学校专业级字段没有来源时不伪造。
- 公司薪资区分官方和估算。
- 官方入口数量、live adapter 数量和有效样本数量口径一致。

工程质量：

- `npm run build` 通过。
- `npm run verify:career-signals` 通过。
- `npm run verify:miniprogram` 通过。
- 新能力优先落在 `src/lib/*` 领域层，避免继续膨胀 `src/main.tsx`。
